  Question 2: How is the short URL identifier generated вҖ” random, or derived

вҸә ADR-001 created at docs/architecture/adr/ADR-001-random-short-id.md.

  Question 3: What's the ID format and length?

  This determines your DynamoDB partition key shape and collision-handling
  logic. Options:
  - Base62 (0-9A-Za-z), URL-safe with no special encoding needed.
  - Base58 (Bitcoin-style вҖ” drops 0, O, I, l) to avoid visually-ambiguous
  characters when read aloud or hand-typed.

  And length: 7 chars of base62 gives ~3.5 trillion combinations; 8 gives ~218
  trillion. At any reasonable pastebin scale (millions to low billions of
  pastes), 7 chars keeps birthday-paradox collision odds effectively zero (need
  ~2M pastes before even a 1-in-a-million chance of one collision at 7 chars)
  while keeping URLs short.

  My recommendation: base62, 7 characters, with a collision check on write
  (re-roll on conflict вҖ” at this ID space, you'll essentially never hit one, but
  the check is cheap insurance). Skip base58's ambiguity-avoidance вҖ” pastebin
  URLs are copy-pasted and clicked, not read aloud or transcribed by hand, so
  the extra restriction isn't buying you anything.

  Agree, or do you want more headroom in the ID space?

вҸә DynamoDB vs. PostgreSQL vs. a couple of others, for the Paste store:

  DynamoDB
  - вң… Fully serverless вҖ” no instance to provision or patch; pairs naturally
  with a serverless/Lambda FastAPI deployment.
  - вң… Point-lookup-by-ID is exactly its sweet spot: single-digit-ms latency at
  any scale, and your access pattern (write once, read by ID) is the textbook
  case DynamoDB is designed for.
  - вң… Built-in TTL attribute вҖ” free win if/when you ever want expiring pastes.
  - вң… Multi-AZ durability by default, satisfies "persistent" out of the box.
  - вҡ пёҸ 400KB item size limit вҖ” since paste text lives in the item, a large paste
  (a big log dump, a whole file) could blow this. Workaround: store body in S3,
  keep only metadata in DynamoDB вҖ” clean, but it's a second moving part.
  - вҡ пёҸ No ad-hoc querying вҖ” fine today (you only ever fetch by ID), but if you
  later want "list my recent pastes," search, or an admin view, you're adding
  GSIs and thinking in access-pattern terms up front rather than just writing a
  SQL query.
  - вҡ пёҸ Less familiar local dev loop (DynamoDB Local / LocalStack vs. just
  running Postgres).

  PostgreSQL (e.g., RDS or Aurora Serverless v2)
  - вң… Flexible querying for free вҖ” "list recent pastes," search, admin tooling,
  analytics all just work later without a storage redesign.
  - вң… No practical size limit on the text column (TOAST handles large text,
  ~1GB/field).
  - вң… Mature FastAPI tooling (SQLAlchemy/SQLModel, Alembic migrations), easier
  local dev.
  - вҡ пёҸ Still an instance/cluster to manage, even "serverless" Aurora has
  cold-start and scaling behavior that's slower and less hands-off than
  DynamoDB.
  - вҡ пёҸ No built-in caching layer вҖ” you'd add ElastiCache/Redis explicitly
  (though realistically you'd want an explicit cache in front of either DB,
  given CLAUDE.md's "cached for fast access" requirement).

  Other options, briefly: MongoDB Atlas / Cassandra-style stores solve the same
  problem as DynamoDB with more ops burden and no AWS-native integration вҖ” not
  worth it here. Using S3 alone as the primary store (keyed by the random ID) is
  actually plausible for content this simple, but you'd still want a fast
  index/metadata layer in front of it, which reintroduces DynamoDB or Postgres
  anyway.

  My recommendation: DynamoDB. Your actual requirement вҖ” durable storage, fast
  point-read by ID, no anticipated joins/search вҖ” is precisely what it's built
  for, and it keeps the whole stack serverless end-to-end (FastAPI on Lambda,
  DynamoDB, S3+CloudFront), which minimizes what you have to operate. The 400KB
  item cap is the one real risk; worth deciding now whether you want a paste
  size limit under that, or a size-based fallback to S3.

  Do you want to go with DynamoDB, or does the flexibility of Postgres (e.g.,
  for a future "my pastes" list or search) outweigh that for you?

  just an env-var-driven endpoint override). This is what you run FastAPI
  against locally, and what CI spins up as a service container for integration
  tests that need real DynamoDB semantics (conditional puts for your collision
  check, etc.).

  One caveat: DynamoDB Local doesn't actively enforce TTL expiry (it accepts the
  attribute but won't background-delete expired items) вҖ” irrelevant today since

  test expiry behavior specifically (would require hitting real AWS for that one

вҸә Three real shapes to consider for the frontend, given Pastes are immutable:

  A. Pure React SPA (your original proposal) вҖ” S3 + CloudFront static hosting, 
  API calls to FastAPI for content
  - вң… Simple, matches the stack you started with, clean frontend/backend
  separation.
  - вҡ пёҸ Viewing a paste means: load HTML shell вҶ’ download JS bundle вҶ’ JS runs вҶ’
  fetch API вҶ’ render text. Several round trips before the user sees anything
  they pasted вҖ” a real cost for something whose whole point is "load fast."
  - вҡ пёҸ Important gap: link unfurlers (Slack, iMessage, Twitter/X previews) fetch
  the raw HTML of a shared URL вҖ” they don't execute JS. A pure SPA view-page
  will unfurl as an empty shell, not the pasted content. For a link-sharing
  tool, that's a real feature loss.
  - вҡ пёҸ Also worth naming: CloudFront caching a SPA mostly caches the static 
  JS/CSS/HTML shell, not the actual paste content вҖ” that's still fetched
  dynamically from the API on every view unless you separately configure caching
  for the API responses too.

  B. Server-side rendering (e.g., FastAPI + Jinja2, or Next.js) for the 
  view-paste route
  - вң… Full HTML with the paste content embedded on first response вҖ” instant
  content, no loading spinner, works with JS off, and unfurlers see the real
  text.
  - вң… Because Pastes are immutable (ADR-001), this rendered HTML can be cached
  at CloudFront forever, with zero invalidation logic вҖ” you get SSR's UX
  benefits and static-cache economics simultaneously.
  - вҡ пёҸ More moving parts: rendering now lives in your compute layer instead of
  being pure static files; adds template/rendering code to maintain.

  C. Render-once-at-write-time (static generation on submit)
  - Since content is known and frozen at creation, FastAPI could render the
  paste's HTML page once, at submit time, and store it as a static object (e.g.
  in S3) rather than rendering per-request. "Viewing" a paste becomes pure
  static file serving вҖ” zero origin compute ever, for the entire lifetime of the
  paste.
  - вң… The most literal way to get "caching for free" вҖ” there's no dynamic
  rendering to cache, it's just a file.
  - вҡ пёҸ Couples paste storage to two systems (DynamoDB for metadata/API access,
  S3 for the rendered page) вҖ” more to keep in sync, though since it's write-once
  this is low-risk.

  A caching subtlety that applies regardless of which you pick: since paste
  content never changes, the API's GET /pastes/{id} response can carry
  Cache-Control: public, max-age=31536000, immutable. If CloudFront also fronts
  the API (not just the S3 static assets), it will cache the actual paste 
  content at the edge indefinitely вҖ” this is what actually delivers on
  CLAUDE.md's "cached for fast repeated access," more than caching the SPA shell
  alone would.

  My recommendation: keep the SPA for the create/compose flow (it's genuinely
  interactive, SPA is the right tool), but for the view-paste route, go with B 
  (SSR) rather than A or C вҖ” C's extra storage system isn't worth it when SSR +
  aggressive Cache-Control gets you the same caching economics with one fewer
  moving part, and B closes the link-unfurl gap that matters a lot for a
  pastebin. That does mean the "view" route needs a small rendering layer in
  your backend rather than being pure static S3 вҖ” but it's just Jinja2/HTML, not
  a second framework.

  Does link-preview/unfurling matter to you for this product? That's the
  deciding factor between A and B.

вҸә Is it common to pair a JS rendering framework with a Python backend? Yes вҖ” but
  it means running two runtimes.

  The common industry pattern (many companies do this) is: Python API backend +
  a separate Next.js service that does SSR by calling that API вҖ” effectively a
  "backend-for-frontend." It's a legitimate, well-trodden pattern. But it's
  important to be precise about what it costs you here: Next.js SSR needs a live
  Node.js process (or Node-based serverless functions) rendering on every cache
  miss, deployed and operated alongside FastAPI вҖ” two languages, two dependency
  trees, two deploy pipelines, two runtimes to monitor, for what is otherwise
  one small backend.

  Jinja2 (via FastAPI's built-in Starlette templating) keeps everything in one
  Python service вҖ” no second runtime, no second deploy target, one CI pipeline
  for the whole backend.

  The deciding factor is what the view-paste page actually needs to do: render
  some (possibly syntax-highlighted) immutable text, with a couple of Open Graph
  meta tags for unfurling, maybe a "copy to clipboard" button. That's it вҖ” no
  client-side state, no interactivity that benefits from React's component
  model. Next.js earns its complexity when you need shared component logic
  between a rich interactive UI and its server-rendered version; this page
  doesn't have that shape.

  My recommendation: Jinja2 in FastAPI. It matches the actual complexity of the
  page, avoids standing up a second language/runtime for the whole project, and
  doesn't foreclose anything вҖ” if the future mutable-paste version wants richer
  interactivity (inline editing, live preview), you can progressively enhance
  the Jinja2 page with vanilla JS or even mount a small React "island" into it
  later, without needing Next.js's full SSR machinery. Reserve React for where
  it already earns its keep: the create/compose flow.

  One consequence worth flagging: this makes your frontend two separate things вҖ”
  a React SPA (build step, S3-hosted, for /) and server-rendered Jinja2 pages
  (for /{paste_id}), rather than one unified frontend codebase. Are you
  comfortable with that split, or does having "one frontend technology" matter
  enough to you that Next.js's operational cost is worth paying?

вқҜ i want it to be jinja2 only without the spa . will it be easy to add

вҸә Yes вҖ” easier than with an SPA, not harder.

  Basic mutability just needs an edit flow: GET /{paste_id}/edit renders a form
  pre-filled with the current content (fetched from DynamoDB), POST 
  /{paste_id}/edit updates the item and redirects back to the view page. That's
  a plain server-rendered form-and-redirect вҖ” the same pattern the create flow
  already uses. No client-side state management, no API layer for the frontend
  to call, no build step. This is exactly the kind of CRUD that server-rendered
  apps have always done well; if anything, an SPA would make this harder (you'd
  need client state, optimistic updates, a JSON API contract) for no benefit
  here.

  The actual hard part of "adding mutability" isn't the frontend at all вҖ” it's 
  the caching model, and that's true regardless of Jinja2 vs SPA. Right now the
  plan is Cache-Control: immutable, max-age=31536000 on paste views, since
  content never changes вҖ” that's what lets CloudFront cache a paste forever with
  zero invalidation logic. The moment a paste becomes editable, that assumption
  breaks: you'd need either a CloudFront invalidation call on every edit, or
  drop immutable/shorten the TTL for editable pastes (ETag-based revalidation),
  or version the URL (/{paste_id}/v2) so each version is still immutable and the
  current pointer is what changes. That's a real design decision to make when
  you get there вҖ” but it's orthogonal to the rendering technology, so dropping

вқҜ base62, 7-character random ID, with a collision check on write. sounds good.  
  explain the flow of collision check on write                                  

вҸә The flow relies on DynamoDB's conditional writes, so it's atomic вҖ” no separate
  read-then-write race condition:

  1. Generate a random 7-char base62 ID.
  2. PutItem the new Paste with a condition expression:
  attribute_not_exists(paste_id). This tells DynamoDB "only write this if no
  item with this partition key already exists" вҖ” checked and written in a single
  atomic server-side operation.
  3. If the item didn't exist, the write succeeds and you're done.
  4. If it did exist, DynamoDB rejects the write with a
  ConditionalCheckFailedException вҖ” that's your collision signal. Generate a new
  random ID and retry the same conditional PutItem.
  5. Cap retries (e.g., 3вҖ“5 attempts) and treat exhausting them as a hard
  failure. At a 7-char base62 space (~3.5 trillion values), this should never
  actually happen in practice for any realistic scale вҖ” the retry loop is
  defensive insurance, not an expected code path.

  The important part: because the existence-check and the write happen as one
  atomic DynamoDB operation, two concurrent requests that happen to generate the
  same random ID at the same instant can't both succeed вҖ” DynamoDB guarantees
  exactly one PutItem wins and the other gets the conditional-check failure and
  retries with a new ID. No locking, no separate "reservation" table needed.

  That closes out the ID generation piece. Next open item: DynamoDB's 400KB item
  size limit means we need a decision on paste size вҖ” is there a max length for
  submitted text, and if someone tries to paste something larger, do we reject
  it outright or fall back to storing the body in S3?

вқҜ explain AWS WAF , compare to aws ALB explain benefits and cost which fits     
  better for this project ?                                                     

вҸә These aren't actually competing options вҖ” they solve different problems, and 
  WAF needs to attach to something else to work.

  ALB's job: Layer 7 load balancing вҖ” distributes requests across your Fargate
  tasks, does health checks, TLS termination, path/host-based routing. It has no
  concept of "this IP is abusing me" вҖ” it will forward every single request it
  receives to a healthy target, as fast as they arrive. It gives you zero abuse
  protection on its own.

  WAF's job: a rules engine that inspects requests before they're allowed to
  reach an origin, and it attaches to either CloudFront (edge) or an ALB
  (regional) вҖ” it isn't standalone infrastructure. Relevant capabilities:
  - Rate-based rules: track request rate per source IP over a rolling 5-minute
  window, auto-block IPs that exceed a threshold. This is the actual
  abuse-prevention mechanism.
  - Managed rule groups (AWS-provided, e.g. SQLi/XSS/IP-reputation lists):
  defense-in-depth, but not very relevant here since DynamoDB access goes
  through the SDK, not raw queries built from user input.
  - Bot Control (separate paid add-on): more sophisticated bot
  fingerprinting/CAPTCHA challenges вҖ” overkill for launch.

  Where to attach it: CloudFront, not the ALB. It's already your front door in
  this architecture, so blocking abusive requests at the edge вҖ” before they
  reach the ALB, Fargate, or DynamoDB вҖ” is strictly better than filtering one
  hop later.

  Cost: WAF is priced per Web ACL ($5/mo) + per rule ($1/mo per rule, a
  rate-based rule is one rule) + $0.60 per million requests inspected. At
  hobby/low-moderate traffic (~1M requests/month), that's roughly $6-7/month 
  total. The ALB itself isn't an incremental cost of choosing WAF вҖ” you're
  already paying for it (~$16/mo base + usage) because Fargate needs it as
  CloudFront's origin per ADR-005, regardless of WAF.

  So the real comparison isn't "WAF vs ALB" вҖ” it's "~$6-7/month for a rate-based
  rule at the edge" vs. "nothing stops a scripted flood from hitting Fargate
  and writing garbage into DynamoDB." Given DynamoDB write costs scale with
  abuse and Fargate would need to scale out to absorb it, that's cheap
  insurance.

  Recommendation: WAF on CloudFront, one rate-based rule on the create-paste
  path (e.g., >100 requests/5min/IP вҶ’ temporary block). Skip managed rule groups
  and Bot Control for launch.

  Does that settle it, or do you still want Flask-Limiter as a second, app-level
  layer of defense too?
