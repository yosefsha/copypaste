‚Źļ What "Static assets & CDN" (phase 5) means

  Looking at TASKS.md:51-55 and ADR-005, the app has two kinds of responses:
  - Dynamic, per-request (create form, view a paste, htmx fragments) ‚ÄĒ these hit
  Flask on ECS Fargate, because they need DynamoDB reads/writes.
  - Static, identical-for-everyone (style.css, pygments.css) ‚ÄĒ these never
  change per request.

  Phase 5 is about giving CloudFront two origins: S3 for the static files, and
  the Fargate ALB (from ADR-005) for everything else ‚ÄĒ so CloudFront routes by
  path/extension to the right backend.

  Why this matters even with SSR + Jinja2

  SSR (ADR-003) is about how HTML is rendered (server-side, no SPA/client
  framework) ‚ÄĒ it says nothing about how CSS/JS files are delivered. Those are
  separate concerns:

  - Right now, Flask serves style.css/pygments.css itself via its default
  static-file handler (src/copypaste/static/). Every request for that CSS is a
  round-trip to a Fargate container, even though the file never changes.
  - Serving unchanging bytes from S3+CloudFront edge caches instead means: fewer
  requests hit Fargate (less compute to scale/pay for), lower latency for users
  (served from the nearest edge location instead of one ALB region), and it
  frees Fargate to focus purely on the DynamoDB-backed dynamic work it actually
  needs to do.

  So it's an infra/performance optimization layered on top of SSR, not a
  contradiction of it ‚ÄĒ the rendering model stays server-side; only the delivery
  path for the non-dynamic bytes changes.

  Keeping static files alongside src/

  Yes ‚ÄĒ and that's already the current setup (src/copypaste/static/pygments.css,
  style.css) using Flask's default static folder, which is exactly what
  TASKS.md:53 already decided ("plain CSS via Flask's default static folder...
  no bundler"). That doesn't need to change for dev/local.

  What phase 5 adds is a deploy-time step: as part of CI/CD (phase 8), sync
  src/copypaste/static/* to the S3 bucket so CloudFront can serve it from there
  in production. The repo's source of truth stays under src/; S3 just becomes a
  deployment target, not a second place you hand-edit files.
